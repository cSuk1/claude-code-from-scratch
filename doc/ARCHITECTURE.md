# Claude Code Mini 架构分析文档

> 生成日期: 2026-04-03
> 项目版本: 1.0.0

## 一、项目概述

Claude Code Mini 是一个轻量级的 TypeScript CLI 编码助手，模仿 Claude Code 的核心架构设计。项目采用 ESM 模块规范，TypeScript 严格模式，编译目标为 ES2022。

### 技术栈

- **语言**: TypeScript 5.8
- **运行时**: Node.js (ESM)
- **核心依赖**:
  - `@anthropic-ai/sdk` - Anthropic API SDK
  - `openai` - OpenAI 兼容 API SDK
  - `chalk` - 终端颜色
  - `glob` - 文件匹配

### 代码规模

| 模块 | 文件数 | 主要功能 |
|------|--------|----------|
| `src/core/` | 6 | 核心代理逻辑、模型管理、压缩管道 |
| `src/tools/` | 5 | 工具定义、执行器、权限系统 |
| `src/cli/` | 4 | 命令行解析、REPL、命令注册 |
| `src/ui/` | 1 | 终端 UI、Markdown 渲染 |
| `src/extensions/` | 2 | 技能系统、子代理系统 |
| `src/storage/` | 2 | 会话持久化、记忆系统 |
| `src/utils/` | 1 | YAML frontmatter 解析 |

## 二、核心架构

### 2.1 入口流程

```
src/cli.ts
    ↓
parseArgs()        → 解析命令行参数
    ↓
resolveApiConfig() → 解析 API 配置 (Anthropic/OpenAI)
    ↓
initModelTiers()   → 初始化三层模型系统
    ↓
new Agent()        → 创建代理实例
    ↓
prompt ? chat() : runRepl()  → 单次对话或交互式 REPL
```

### 2.2 Agent 核心类 (src/core/agent.ts)

Agent 是系统的核心，约 1070 行代码，负责：

1. **双后端支持**: 同时支持 Anthropic 和 OpenAI 兼容 API
2. **消息历史管理**: 维护两套独立的消息格式
3. **工具执行循环**: 流式处理 → 工具调用 → 权限检查 → 执行 → 结果返回
4. **Token 追踪**: 输入/输出 token 统计
5. **上下文压缩**: 四层压缩管道
6. **子代理支持**: 独立的子代理实例管理

### 2.3 三层模型系统 (src/core/model-tiers.ts)

```
┌─────────────────────────────────────────────────────────┐
│                     PRO Tier                            │
│  主对话、复杂推理、general 代理                           │
│  默认: glm-5                                            │
├─────────────────────────────────────────────────────────┤
│                     LITE Tier                           │
│  子代理、explore、plan 任务                               │
│  默认: minimax-m2.5                                     │
├─────────────────────────────────────────────────────────┤
│                     MINI Tier                           │
│  快速查询、compact 压缩                                   │
│  默认: kimi-k2.5                                        │
└─────────────────────────────────────────────────────────┘
```

**配置优先级** (高→低):
1. 运行时命令: `/model pro <name>`
2. 环境变量: `MINI_CLAUDE_MODEL_PRO` / `_LITE` / `_MINI`
3. 配置文件: `.ccmini/settings.json`
4. 内置默认值

### 2.4 上下文压缩管道 (4 层)

| 层级 | 名称 | 触发条件 | 操作 | API 调用 |
|------|------|----------|------|----------|
| 1 | Budget | 利用率 > 50% | 截断大工具结果 (保留头尾) | 否 |
| 2 | Snip | 利用率 > 60% | 用占位符替换旧/重复结果 | 否 |
| 3 | Microcompact | 空闲 > 5 分钟 | 激进清理旧结果 | 否 |
| 4 | Auto-compact | 利用率 > 85% | API 生成对话摘要 | 是 |

## 三、工具系统

### 3.1 内置工具 (7 个)

| 工具名 | 类别 | 功能 |
|--------|------|------|
| `read_file` | 读取 | 读取文件内容，带行号 |
| `list_files` | 读取 | Glob 模式匹配文件列表 |
| `grep_search` | 读取 | 正则搜索文件内容 |
| `write_file` | 写入 | 创建/覆盖文件 |
| `edit_file` | 写入 | 精确字符串替换编辑 |
| `run_shell` | 执行 | 执行 shell 命令 |
| `skill` | 扩展 | 调用注册的技能 |
| `agent` | 扩展 | 启动子代理 |

### 3.2 权限系统 (src/tools/permissions.ts)

**5 种权限模式**:

| 模式 | 描述 |
|------|------|
| `default` | 默认模式，危险操作需确认 |
| `plan` | 只读模式，禁止写入操作 |
| `acceptEdits` | 自动接受编辑操作 |
| `bypassPermissions` | 跳过所有权限检查 |
| `dontAsk` | 自动拒绝需确认的操作 |

**危险命令检测** (正则匹配):
- 文件删除: `rm`, `del`, `rmdir`
- 系统操作: `sudo`, `mkfs`, `dd`, `reboot`
- 进程控制: `kill`, `pkill`, `taskkill`
- Git 危险操作: `git push/reset/clean`

**权限规则持久化**:
```json
// .ccmini/settings.json
{
  "permissions": {
    "allow": ["run_shell(npm test*)", "write_file(src/**)"],
    "deny": ["run_shell(rm *)"]
  }
}
```

## 四、扩展系统

### 4.1 子代理系统 (src/extensions/subagent.ts)

**内置代理类型**:

| 类型 | 工具集 | 用途 |
|------|--------|------|
| `explore` | 只读工具 | 快速代码库探索 |
| `plan` | 只读工具 | 结构化实现规划 |
| `general` | 全部工具 (除 agent) | 独立任务执行 |
| `compact` | 无工具 | 对话压缩 (内部) |

**自定义代理**:
```markdown
<!-- .ccmini/agents/code-reviewer.md -->
---
name: code-reviewer
description: 代码审查代理
allowed-tools: read_file, grep_search, list_files
model: lite
---
你是一个代码审查专家...
```

### 4.2 技能系统 (src/extensions/skills.ts)

**技能定义格式**:
```markdown
<!-- .ccmini/skills/test-gen/SKILL.md -->
---
name: test-gen
description: 生成单元测试
when-to-use: 需要为函数生成测试时
user-invocable: true
context: inline | fork
allowed-tools: read_file, write_file
model: lite
---
为以下代码生成单元测试:
$ARGUMENTS
```

**执行模式**:
- `inline`: 注入提示到当前对话
- `fork`: 在隔离子代理中执行

### 4.3 记忆系统 (src/storage/memory.ts)

**4 种记忆类型**:

| 类型 | 用途 | 示例 |
|------|------|------|
| `user` | 用户偏好、知识水平 | "用户是高级开发者" |
| `feedback` | 用户纠正和指导 | "总是使用函数式组件" |
| `project` | 项目目标、决策 | "使用 PostgreSQL 数据库" |
| `reference` | 外部资源指针 | "API 文档: docs.example.com" |

**存储位置**: `~/.ccmini/projects/<hash>/memory/`

**自动索引**: 写入记忆时自动更新 `MEMORY.md`

## 五、用户界面

### 5.1 REPL 命令

| 命令 | 描述 |
|------|------|
| `/help` | 显示所有命令 |
| `/model [tier] [name]` | 查看/切换模型 |
| `/clear` | 清空对话历史 |
| `/compact` | 手动压缩对话 |
| `/cost` | 显示 token 使用量 |
| `/memory` | 列出已保存记忆 |
| `/skills` | 列出可用技能 |

### 5.2 Markdown 渲染

- 代码块: 带语言标签的边框样式
- 表格: Unicode 边框绘制
- 标题: 颜色/粗细区分
- 列表: 圆点/数字样式
- 链接: 下划线 + URL 提示

### 5.3 交互式菜单

- 方向键导航
- Enter 确认
- Ctrl+C / Escape 取消
- 权限确认选项: 允许/允许并记住/拒绝/拒绝并记住

## 六、改进建议

### 6.1 功能增强

#### 1. 测试框架
**现状**: 未配置测试框架
**建议**: 
- 添加 Jest 或 Vitest
- 为核心模块编写单元测试
- 添加集成测试覆盖工具执行

#### 2. 日志系统
**现状**: 仅 `console.log` 输出
**建议**:
- 添加可配置的日志级别 (debug/info/warn/error)
- 支持日志文件输出
- 添加请求/响应追踪 (便于调试 API 问题)

#### 3. 插件系统
**现状**: 技能和代理通过文件系统发现
**建议**:
- 支持 npm 包形式的插件
- 添加插件生命周期钩子
- 提供插件 API 文档

#### 4. 多语言支持
**现状**: UI 文本硬编码英文
**建议**:
- 添加 i18n 支持
- 支持中文界面

### 6.2 架构优化

#### 1. 错误处理
**现状**: 部分 try-catch 静默忽略错误
**建议**:
```typescript
// 当前
catch { /* skip */ }

// 改进
catch (e) {
  logger.debug(`Failed to load ${file}: ${e.message}`);
}
```

#### 2. 类型安全
**现状**: 部分使用 `any` 类型
**建议**:
- 为工具输入定义严格的接口类型
- 避免类型断言，使用类型守卫

#### 3. 配置验证
**现状**: 配置文件解析无验证
**建议**:
- 使用 Zod 或 JSON Schema 验证配置
- 提供友好的配置错误提示

#### 4. 模块解耦
**现状**: Agent 类职责较多 (1070 行)
**建议**:
- 抽取工具执行逻辑到独立类
- 分离 Anthropic/OpenAI 后端实现
- 使用依赖注入

### 6.3 性能优化

#### 1. 文件监控缓存
**现状**: 每次请求重新发现技能/代理
**建议**:
- 使用 chokidar 监控文件变化
- 仅在文件变化时重新加载

#### 2. 并行工具执行
**现状**: 工具顺序执行
**建议**:
- 检测无依赖的工具调用
- 并行执行独立工具

#### 3. 流式输出优化
**现状**: 每个字符单独 write
**建议**:
- 批量缓冲输出
- 减少TTY 写入次数

### 6.4 安全增强

#### 1. 敏感信息过滤
**现状**: 工具结果直接输出
**建议**:
- 检测并过滤 API key、密码等敏感信息
- 提供敏感信息脱敏选项

#### 2. 沙箱执行
**现状**: shell 命令直接执行
**建议**:
- 可选的 Docker 沙箱模式
- 限制文件系统访问范围

#### 3. 审计日志
**现状**: 无操作记录
**建议**:
- 记录所有工具调用
- 支持审计日志导出

### 6.5 开发体验

#### 1. 调试模式
**现状**: 有限的调试信息
**建议**:
- `--debug` 标志启用详细日志
- 显示 API 请求/响应详情
- 显示 token 计算过程

#### 2. 配置向导
**现状**: 需手动创建配置文件
**建议**:
- 首次运行时交互式配置
- 自动检测环境变量

#### 3. 文档完善
**现状**: README 和 CLAUDE.md
**建议**:
- 添加 API 文档
- 添加贡献指南
- 添加更多使用示例

## 八、总结

Claude Code Mini 是一个设计精良的轻量级编码助手，核心特性包括：

**优点**:
- 清晰的模块划分和代码组织
- 双后端支持 (Anthropic/OpenAI)
- 完善的三层模型系统
- 四层上下文压缩管道
- 灵活的技能和子代理扩展机制
- 细粒度权限控制

**待改进**:
- 缺少测试框架
- 错误处理和日志系统可增强
- Agent 类职责较重，可进一步拆分
- 缺少插件生态系统

项目整体架构合理，代码质量较高，适合作为学习现代 CLI AI 工具设计的参考项目。
